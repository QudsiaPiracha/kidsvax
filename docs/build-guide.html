What is KidsVax?

In Germany, every child has two paper booklets: the Gelbes Heft (yellow booklet for pediatric checkups) and the Impfpass (vaccination passport). Parents must track 14 checkups, 6 dental exams, and dozens of vaccinations across these two booklets — in German only.

KidsVax digitizes both into a single web app. It automatically generates schedules based on each child's birthday, sends email reminders, tracks growth with charts, and even lets parents photograph their paper records so AI can extract the data. It's bilingual (English default, German) and built for the German market.

Key Point for Beginners

We built this entire application by having conversations with Claude Code — an AI coding assistant that runs in your terminal. We never opened a code editor. We described what we wanted in plain English, and Claude Code wrote all the code, ran all the tests, set up the database, and deployed it.

Phase 1 — Writing the Specification

Every good project starts with a clear description of what you're building. We started by telling Claude Code about our idea, and it helped us create a detailed specification document.

1

Describe Your Idea

We described KidsVax: a mobile-friendly web app to digitize Germany's Gelbes Heft and Impfpass. We explained the problem (paper booklets get lost, no reminders, German-only), the target users (parents in Germany), and the features we wanted (auto schedules, AI scanning, growth charts, reminders).
Claude Code created SPEC.md — a 1,000+ line specification covering: every feature in detail, the database design, API endpoints, UI mockups in text, the tech stack decision, and a competitive analysis table comparing KidsVax to existing apps.
2

Create the TODO List

We asked Claude Code to break the spec into actionable tasks with test criteria.
Claude Code created 17 detailed TODOs in _project_specs/todos/active.md, each with: a description, acceptance criteria, test cases to write, and dependencies on other tasks. These became our development roadmap.
Tip: Be Specific About What You Want

The more detail you give Claude Code about your idea, the better the spec will be. Mention your target users, what makes your app different, any regulations you must follow (like Germany's Masernschutzgesetz), and your preferred technologies if you have any.

With the spec ready, we told Claude Code to set up the actual project. This is where code starts being written.

Claude Code created:

How TDD Worked in Practice

For each feature, Claude Code first wrote test files describing what the code should do. Then it ran the tests (they failed — this is expected, called "RED"). Then it wrote the actual code. Then it ran the tests again (they passed — "GREEN"). This ensures every feature works correctly and nothing breaks when new code is added. We ended up with 441 tests across 31 test files.

Phase 6 — Fixing Deployment Errors

Things don't always work on the first try. Here are the real errors we hit and how Claude Code fixed them.

!

Error 1: Next.js 15 Named Exports

"Also fix deployment errors on Vercel: 'ForgotPasswordPage' is not a valid Page export field."
Next.js 15 doesn't allow named exports alongside export default in page files. Claude Code changed export function ForgotPasswordPage to just function ForgotPasswordPage in all auth pages. Same fix applied to LoginPage, SignupPage, and layout files.
!

Error 2: Promise Params in Next.js 15

Next.js 15 changed how dynamic route parameters work — they're now Promises that must be awaited. Claude Code updated all [id] pages from params: { id: string } to params: Promise<{ id: string }> and added await.
!

Error 3: Security Vulnerability Block

Vercel blocked deployment because Next.js 15.1.0 had a known security vulnerability (CVE-2025-66478). Claude Code upgraded to Next.js 15.5.12, ran all 441 tests (still passing), and redeployed successfully.
This is Normal

Even experienced developers hit deployment errors. The key is that Claude Code can read error messages, understand what's wrong, and fix them — often faster than a human developer could. We hit 4 different error types and Claude Code fixed all of them within minutes.

Lessons Learned

1. Start with a Detailed Spec

The better your initial description, the better everything downstream. Our SPEC.md was 1,000+ lines and covered every feature, edge case, and design decision. This meant Claude Code rarely had to guess.

2. Break Work into Small, Testable TODOs

17 focused tasks were easier to build than one giant "build the app" instruction. Each task had clear acceptance criteria and test cases, so Claude Code knew exactly when it was done.

3. Use Parallel Teams for Speed

By splitting work into two teams (Alpha for backend, Beta for frontend) and using git worktrees for isolation, we built features in parallel without merge conflicts. This dramatically reduced build time.

4. TDD Catches Bugs Early

Writing tests first meant every feature was verified as soon as it was built. When we upgraded Next.js or changed code, we could immediately confirm nothing broke by running npm test — all 441 tests passed every time.

5. Deployment Always Has Surprises

Code that works locally can fail in production. We hit 4 different deployment errors (named exports, Promise params, type mismatches, security CVE). Claude Code fixed all of them, but expect this phase to take some back-and-forth.

6. Test with Real Data

An empty dashboard doesn't tell you if the app works. We seeded two test children with realistic medical data (completed exams, overdue vaccinations, growth measurements) to properly evaluate the experience.
